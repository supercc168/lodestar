# GSD 规划效率与质量策略

本策略用于减少 GPT-5.6 已具备的规划能力与 GSD 研究/规划/检查链之间的重复工作。目标是降低无收益的调用与推理，不降低复杂阶段的规划、审查、恢复和验收质量。

## 不可牺牲项

- `TRACKER.md`、`TASK.md`、`STATE.md`、跨平台 workstream 路由、单调游标和 `.gsd` 本地提交规则保持不变。
- 标准 Phase 计划必须由 `gsd-planner` 产出，并由 `gsd-plan-checker` 独立检查；禁止默认传入 `--skip-verify`。
- 子 agent 继续执行 `extra-codex-agent-policy.md`：planner / researcher 等标准或重型职责使用 `high`，plan-checker 按官方 routing tier 使用 `medium`。
- 实现后的代码审查、测试、验证和 `extra-finalization-gate.md` 终验代际门禁不因本策略减少。
- 主 Codex 的模型和推理强度由用户当前会话决定，本 skill 不修改用户级 `config.toml`。

## 路由顺序

进入创建任务、讨论、计划、继续或自动推进前，按以下顺序判断：

1. 从 `TRACKER.md`、当前 `TASK.md`、`STATE.md` 和已有阶段产物确认唯一下一动作；禁止为了效率重新规划已是 GREEN / 已验证的工作。
2. 若已有 `RESEARCH.md`，先判断它是否覆盖当前 Phase 的完成标准、依赖和代码事实；覆盖且没有失效证据时直接复用。
3. 没有可复用研究时，按“轻量计划判定”决定是否在内部调用标准计划的 `--skip-research`。
4. 不满足全部轻量条件时走标准计划，让原生 GSD 按配置/交互决定研究；高风险条件命中时必须研究。
5. 除非命中“重型入口授权”，禁止额外叠加 Spec、Power Discuss、Convergence、Ultraplan、Chunked、Bounce 或 Autonomous。

## 轻量计划判定

只有以下条件**全部满足**，才允许跳过 Phase Research：

- 需求、范围、完成标准和禁止事项已经写入 `TASK.md` / `CONTEXT.md`，不存在会改变实现方向的灰区。
- 只使用仓库内已确认的实现模式，不引入新框架、新依赖、新外部 API 或陌生工具链。
- 不新增或改变公共架构、包依赖方向、持久化格式、协议、数据迁移、安全边界、并发模型或性能关键路径。
- 相关代码入口、预期修改范围和验证命令已经由当前代码事实确认。
- 当前阶段没有 `[NEEDS DECISION]`、低置信度研究结论、未解决阻断或要求先做技术验证的约束。

只要任一项不满足或无法确认，就不得判为轻量。轻量只代表省略独立 researcher，不代表省略 planner、plan-checker、执行验证或终验。

## 必须研究的信号

命中任一项时走标准研究，不得为提速跳过：

- 新架构、跨包/跨程序集职责变化、依赖方向变化或公共扩展点设计。
- 新第三方库、外部服务、协议、生成工具、编辑器能力或版本兼容问题。
- 数据迁移、资源格式、序列化、持久化、安全、并发、热路径性能或平台差异。
- 现有代码事实与文档冲突，或关键实现路径仍需猜测。
- 近期代码/需求变化已使已有 `RESEARCH.md` 的关键假设失效。
- `TASK.md`、用户或阶段证据明确要求研究、Spike、对照方案或架构评审。

## 重型入口授权

以下能力均为选择性增强，不属于普通 GSD 下一步：

| 能力 | 允许触发条件 |
|------|--------------|
| Spec Phase | 用户明确要求先锁定规格，或需求无法形成可验证完成标准 |
| Power Discuss | 用户明确要求深挖全部灰区，普通讨论不足以消除关键歧义 |
| Plan Review Convergence | 用户明确要求跨 AI 复审/收敛，或 TASK 明确把它列为交付门禁 |
| Ultraplan | 用户明确要求使用该 Beta/云端规划能力 |
| Autonomous | 用户明确授权连续推进多个阶段 |
| Chunked Plan | 单次 planner 已有可复现的规模/截断失败，或用户明确要求分块 |
| Plan Bounce | 已配置外部 refinement 脚本且用户或 TASK 明确要求 |
| Quick Full | 用户明确要求官方 Quick 全链，且已确认不会违反项目 Git 与 `.gsd` 边界 |

“更全面”“保险起见”“GPT-5.6 能多想”不是授权依据。一次明确授权只适用于用户指定的当前范围，不得自动延续到后续 Phase。

## 主编排去重

- 主编排 agent 在调用 planner 前只整理事实、边界、完成标准和必要文件，不先自行写一份详细实现方案再让 planner 重写。
- planner 返回后以 `gsd-plan-checker` 结论为准；主编排只处理阻断、持久化状态和用户检查点，不额外启动第二套泛化审查。
- checker 要求修订时只传递当前发现，遵循原生最多三轮的定向修订；禁止无新发现地重新研究或重跑完整规划链。
- 已有新鲜产物优先复用。只有代码、依赖、需求或完成标准发生实质变化，才能宣告产物失效，并写明失效依据。

## 官方更新边界

本策略只存在于项目 `.agents/skills/yiui-gsd/` 定制层。禁止为实现本策略修改以下安装器管理区域：

- `~/.agents/skills/gsd-*`
- `~/.codex/gsd-core/`
- `~/.codex/agents/gsd-*.toml`

GSD 更新后：

1. 确认本文件、`SKILL.md` 中的规划效率门禁和 `extra-phrase-map.md` 的路由仍存在。
2. 按 `extra-codex-agent-policy.md` 重放并验证子 agent 策略。
3. 对照新版官方参数检查本策略引用的入口是否仍有效；只修改项目定制层，不在官方文件上打补丁。
4. 用新 Codex 任务验证 skill 发现与路由；当前缓存任务不能证明更新后的 skill 已重新加载。
