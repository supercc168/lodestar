## 1. TL;DR(30 秒建立心智)

- **它是什么**:一个 **Bun daemon**,把**飞书群消息**桥接到**无头 AI Agent 后端**(Claude Code / Codex),让 AI 在本机项目目录里干活,过程用飞书 **Card Kit 卡片**流式回传。
- **一句话架构**:`飞书群消息 → daemon → Session(一个群一个) → AgentProcess(Claude/Codex) → 在项目目录读写代码 → 事件流 → cardkit 画成卡片发回群`。
- **技术栈**:Bun(跑/测/构建)+ Node ≥ 18(发布包运行)+ TypeScript;`@larksuiteoapi/node-sdk`(飞书)、`@anthropic-ai/claude-agent-sdk`(Claude)、`codex` CLI(Codex)。
- **入口**:`cli.ts` → `daemon.ts` → `src/session.ts`(核心)。
- **怎么跑**:`bun daemon.ts`(开发)/ `bun run build`(产物)/ `bun test`(测试)。

## 2. 核心概念与架构

### 2.1 三条铁律(记住就懂一半)
1. **一个飞书群 = 一个项目工作区(`Session`)**,群名映射到 `projects_root` 下同名目录。
2. **一个 Session = 一个选定 provider 的 agent 进程**(Claude 长驻 `query()` 进程,或 Codex `app-server` 进程)。
3. **每轮对话 = 一张流式卡片**,agent 的正文/工具调用/命令/diff/状态都在同一张卡上更新。

### 2.2 运行时数据流
```
飞书 WS 事件(im.message.receive_v1 / card.action.trigger)
   │
   ▼  daemon.ts::handleMessage
去重 → 去过期 → chat_id 映射群名 → sessionFor(chat, name)
   │  (解析 text/post/image/file;附件下载到 inbox/;裸词命令/Ask 拦截)
   ▼  Session.onUserMessage(text, files, ...)
Session(src/session.ts)—— 一个群的状态机
   │  经统一 AgentProcess 接口
   ▼
ClaudeAgentProcess(默认) 或 CodexProcess
   │  agent 在项目目录读写代码、跑命令、调工具
   ▼  发出统一事件(assistant text / tool_use / tool_result / result / compact)
Session 把事件 → cardkit 操作(addElement / replaceElement)
   ▼
飞书卡片实时更新
```

### 2.3 双后端(统一接口)
两类后端都实现 `src/agent-process.ts` 的 `AgentProcess`(`EventEmitter`)接口,`Session` 只依赖抽象:
- **Claude(默认)** `src/claude-agent-process.ts`:`@anthropic-ai/claude-agent-sdk` 的 `query({ prompt: AsyncIterable })` **streaming-input 长驻进程**,`bypassPermissions` 全自动。档位 Fable 5 / Opus 走登录态,GLM 走第三方路由。
- **Codex** `src/codex-process.ts`:`codex app-server --listen stdio://`,JSON-RPC,需 ChatGPT 登录,固定 `gpt-5.5`/`xhigh`。
- 群里 `model` 命令切换,按 session+provider 持久化。
- `sendUserText` 返回 provider 感知的投递句柄:Codex 是 `turn_start_pending`(带永不 reject 的 `settlement` 回执,ACK = 首个精确的 turn/start RPC 响应 / `turn/started` / `turn/completed`);Claude 是同步 `queued`/`rejected`。`Session` 的真人批次只在 ACK 后提交,reject/exit 先于 ACK 则连同附件、顺序、`msgId` 与 reaction 一次性原样回存。

### 2.4 Codex turn watchdog(卡死自动恢复)

运行契约(与 README 一致,改动语义前先读 `docs/archive/specs/2026-07-13-codex-turn-watchdog-auto-recovery-design.md`):

> Codex 默认 `recover_once`。最后一次有效进展后持续 15 分钟,且确认至少 10 次完全相同、成功、无副作用的 `text(...)` 空调用时,才在原 thread 自动恢复一次。纯静默推理只在 30 分钟后提示,不会自动打断;同一任务链第二次确认循环只停止,不再恢复。真人排队消息始终优先并保留附件、顺序与 reaction;Claude 不启用自动恢复。

实现拆分:
- `src/turn-watchdog.ts`:**纯函数检测器**(指纹、阈值、verdict),不做任何 I/O,单测最全。
- `src/session.ts`:生命周期挂钩 —— `beginWatchdogTurn` 装 tick interval,`runWatchdogRecovery`/`runWatchdogExhausted` 执行动作;所有动作使用不可变 `WatchdogActionTransaction` 快照 + lifecycle lease,await 之后一律复核所有权,失败时保留 `preservedWatchdogRecovery`(不可变 provider+threadId),用户 `restart` 恢复排队消息,`kill`/`clear` 显式丢弃。
- 脱离归属的旧卡由 `terminalizeDetachedTurn` 收尾(等 rotating → footer 终态 → streaming off → dispose),任一终态写失败只发一次 raw 文本兜底 —— cardkit 的 `replaceElement`/`patchSettings` 为此支持逐调用失败回调。
- 配置面:`config.toml` `[watchdog]`(`codex_mode`/`stall_seconds`/`repeat_noop_limit`/`silent_warn_seconds`/`interrupt_grace_seconds`)+ 项目级 `watchdog_mode` 覆盖;默认 `recover_once`/900s/10 次/1800s/10s。
- **改代码/配置不会自动重启 daemon**;live 验证需要用户当条消息明确授权的重启(见红线一)。

### 2.5 飞书 GSD 状态卡

群内发裸词 `gsd`(或 `gsd status`,大小写不敏感)打开/刷新 **GSD 状态卡**。面板 **跟随当前群 model 档位**(Claude / Codex,与 `model` 命令同一套 session provider),不另开后端。

| 按钮 | 含义 |
| --- | --- |
| **进度** | 重读磁盘 `.gsd` 快照并刷新面板 |
| **继续** | 已暂停则先恢复;空闲时注入 continue 提示推进下一步;会话忙碌则拒绝且不改 store |
| **暂停** | 仅**运行中**任务可暂停(写 TRACKER) |
| **完成** | 将**运行中/已暂停**任务标为已完成 |
| **新任务** | 进入等名态(约 300s);下一条文本消息作为任务名并激活 |

规划**只走 yiui-gsd**:注入提示会禁止 superpowers / OMC / ralplan / ralph / ultrawork 等旧规划入口。状态以磁盘 `.gsd/TRACKER.md` 与活跃任务 `.planning`(经项目根 bridge)为准,不以聊天计划为准。

实现入口:`src/session-gsd.ts` + `src/cards/gsd.ts` + `src/gsd-store.ts` / `gsd-bridge.ts` / `gsd-prompt.ts`;`daemon.ts` 处理 `gsd_*` 卡片回调。

**跨设备安装 GSD 运行时**(checkout 后):仓库内 skill 会随 git 来,全局 `~/.codex/gsd-core` 与 `gsd-*` 不会。在仓库根执行 `bash install/yiui-gsd/install.sh`(Windows:`install.ps1`),校验 `bash install/yiui-gsd/verify.sh`。详见 `install/yiui-gsd/README.md`。

> 代码变更后需**重启 daemon** 才生效;重启属 live-service,须用户**当条消息明确授权**(见红线一),AI 只报告不主动重启。Claude **全局**清理(`~/.claude` 等本机运维)见 Task 7 步骤,不在本仓库改动范围内。

## 3. 代码地图(从哪读起)

**推荐阅读顺序**(由外到内):
1. `cli.ts`(薄入口:缺配置触发向导,否则加载 daemon)
2. `daemon.ts`(飞书长连接、事件分发、`handleMessage`、裸词命令、附件下载)
3. `src/session.ts`(核心状态机,~3000 行 —— 最重要)
4. `src/agent-process.ts`(统一后端接口)
5. `src/claude-agent-process.ts` / `src/codex-process.ts`(两个后端实现)
6. `src/cardkit.ts` + `src/cards/`(卡片渲染)

**关键模块速查**:

| 模块 | 职责 |
| --- | --- |
| `src/session.ts` | 一个群一个 `Session`:状态、agent 生命周期、入站消息、turn/card 状态机;命令下沉到 `session-*` |
| `src/session-commands.ts` | 裸词命令路由(`hi`/`stop`/`kill`/`restart`/`clear`/`compact`/`model`/`gsd`/`task`/`wt`/`agy`) |
| `src/session-gsd.ts` | 飞书 GSD 状态卡:开卡、按钮动作、panelGen 校验、注入 continue/new-task |
| `src/gsd-store.ts` / `gsd-bridge.ts` / `gsd-prompt.ts` | TRACKER 读写、`.planning` bridge、yiui-gsd 注入提示 |
| `src/cards/gsd.ts` | GSD 状态卡渲染(进度/继续/暂停/完成/新任务) |
| `src/feishu.ts` | 飞书 API:token 缓存、群名映射、消息、reaction、**附件下载**、上传、目录初始化 |
| `src/cardkit.ts` | Card Kit 封装:per-card sequence、队列、限流、写失败检测(终态 `replaceElement`/`patchSettings` 支持逐调用失败回调) |
| `src/turn-watchdog.ts` | 纯函数 Codex 空转检测器:工具调用指纹、15 分钟无进展 + ≥10 次相同成功空调用才给 `recover` 判定;纯静默 30 分钟只 `silent_warn` |
| `src/tasklist-worker.ts` | 任务清单自动化后台 worker |
| `src/worktree.ts` | `wt` 的所有 git worktree 逻辑(**git 命令集中在这里,别散写**) |
| `src/config.ts` | 同步读 `config.toml`(含 `[watchdog]` 全局节与 `[projects.*].watchdog_mode` 覆盖) |
| `src/paths.ts` | 所有 runtime 路径常量(XDG/Windows) |

> 每个目录都有 `AGENTS.md`,进目录前先读它的模块表和规则。

## 4. 开发规则与红线(必须遵守)

这些提炼自 `AGENTS.md`,**违反会造成真实事故**:

### 🔴 红线一:不准擅自碰正在运行的 daemon / user service
- **禁止**为了"测试 / 预览 / 发一张看看 / 先验证"去 stop、restart、切换、shadow、并行接管正在运行的 daemon。
- 只有用户在**当前这一条消息里明确点名**(如"重启 daemon"、`lodestar-stop`、`restart`)才能动手。
- **授权不跨消息生效**:上一条消息说过"提交并重启",这一条只说"继续/再改一下"时,**视为没有重启授权**,要重启先重新征得许可。
- 代码改动后,只**报告需要重启**,不主动重启。

### 🔴 红线二:runtime 状态不入库
- 配置在 `~/.config/lodestar/config.toml`,日志/runtime map 在 `~/.local/share/lodestar/`。
- 凭据只在 `config.toml`,**绝不写进仓库**。新状态文件定义在 `src/paths.ts` 并落到 `DATA_DIR`/`CONFIG_DIR`。

### 🔴 红线三:卡片写操作必须走 `cardkit.ts` 队列
- 不要从 session 或脚本直接 `fetch` 改同一张生产卡(会破坏 sequence)。
- 正文按完整段 `addElement`,footer 状态用 `replaceElement`,**不走 Card Kit 打字流**。
- `card.action.trigger` 需 3 秒内替换 JSON 卡时 return `{ card: { type: "raw", data: newCard } }`;别在回调 ACK 前调 `message.patch`(会竞态闪烁)。

### 🔴 红线四:失败要暴露,不静默兜底
- API 失败要 log 并向用户暴露;**不要**静默切换传输/卡片/消息通道当"兜底"。

### 其它约定
- `wt` 的 git 操作集中在 `worktree.ts`,不在 `session.ts` 散写 `git`。
- `Session` 无 `private` 字段是 package-internal 约定,只应由 `session-*.ts` 访问。
- `model` 是固定档位(codex=gpt-5.5/xhigh、claude 档位 effort 锁死),不动态拉列表。

## 5. 本地开发工作流

```bash
# 运行时是 Bun
bun install                 # 装依赖

# 开发跑 daemon(直接跑源码,不构建)
bun daemon.ts               # 或 bun run start
#  → 需要 ~/.config/lodestar/config.toml 里有 [feishu] 凭据,否则报错

# 构建发布产物(bun build --target=node,生成 dist/ 下 5 个 Node 可执行)
bun run build

# 单元测试
bun test                    # 全量
bun test src/session.test.ts  # 单文件
```

**改代码后的自检(按 `AGENTS.md` 测试要求)**:
- 改核心 TS → `bun test`
- 改构建入口 / CLI / 发布相关 → `bun run build`
- 改 session / Card Kit / 飞书消息 / 权限 / Ask 流程 → 补真实群 smoke(见第 6 章)
- 改 `model` / `wt` 创建解散 → 用 debug 注入在真实群验证,测完清理临时 worktree/群/分支

## 6. 调试指南

### 6.1 单元测试(最快)
```bash
bun test                       # 全量(当前 380+ 用例)
bun test src/<模块>.test.ts     # 定向
```
测试用 `bun:test`(不是 vitest)。注意:有些测试断言"未配置 GLM"的行为,**若本机 config 已配 GLM 会污染**,写这类测试要隔离 `config.claude.models`(见 `session.test.ts` 里 `prev → 置空 → finally 恢复` 的 pattern)。

### 6.2 免飞书调试:debug socket 注入(**开发最常用**)
不碰真实飞书就能把一条消息喂进 `handleMessage` 全链路:
```
① daemon 必须在运行(bun daemon.ts)
② 建 debug context(二选一):
   - 在目标群真发一条 `[DEBUG]你好` 文本(daemon 会 seed chat_id+sender_open_id 到 debug-context.json),或
   - bun scripts/seed-debug-ctx.ts "<群名>"
③ 注入消息:bun scripts/test-inject.ts   # 通过 DEBUG_SOCK_FILE 走真实 handleMessage 路径
```
- `scripts/test-mid-turn-rotation.ts` 复现中途换卡,注入后自动截取当天日志。
- debug socket / context 路径定义在 `src/paths.ts`(`DEBUG_SOCK_FILE` / `DEBUG_CTX_FILE`),文件权限 0600、仅本机。

### 6.3 真实飞书 smoke(会发可见消息)
```bash
bun scripts/smoke.ts "<群名>"      # 最小 smoke:发预告 + 直接调 Session.onUserMessage
bun scripts/test-all.ts "<群名>"   # 全流程:kill/hi/流式工具/[[send:]]/中途消息/入站图片/restart/clear
bun scripts/cardkit-probe.ts       # Card Kit API 真实探针
```
> ⚠️ smoke 用**生产 config.toml + 真实 API**,会触达真实飞书/Codex/本机 state。运行前确保同群没有另一个 daemon session 并发抢占;人工观测的脚本会先发 `🧪` 预告便于对日志。

### 6.4 看日志与运行时状态
| 路径 | 内容 |
| --- | --- |
| `~/.local/share/lodestar/daemon-YYYY-MM-DD.log` | 主日志(按日滚动,保留 7 天) |
| `~/.local/share/lodestar/inbox/` | 入站附件下载落地(`{时间戳}-{名字}`) |
| `~/.local/share/lodestar/*.json` | session/chat/resume/model/alive/tasklist map、pid |
| `~/.config/lodestar/config.toml` | 配置(含凭据) |

> 排查飞书图片"收不到":看 `[file:]` 是不是 `.../inbox/{时间戳}-`、`ls inbox/`、日志有无 `feishu: downloaded image`。**用户粘贴的路径文本 ≠ 真图片附件**。

## 7. 配置与环境

### 7.1 config.toml(`~/.config/lodestar/config.toml`)
| 节 | 字段 |
| --- | --- |
| `[feishu]` | `app_id`、`app_secret`(**必填**,没有 daemon 起不来) |
| `[runtime]` | `projects_root`(默认主目录) |
| `[notify]` | `bind`(默认 127.0.0.1)、`port`(默认 9876) |
| `[claude]` | `default_model`、`bin`(自定义 claude 可执行,如 reclaude)、`default_setting_sources` |
| `[claude.models.<名>]` | 第三方路由(GLM):`base_url`/`auth_token`/`model`/`effort` |
| `[projects.<群名>]` | 项目隔离:`cwd`/`setting_sources`/`strict_mcp`/`tools`/`load_project_mcp` |

飞书自建应用怎么建、开哪些权限(尤其 `im:message.group_msg`)、事件订阅长连接、发布版本 —— 见 `lodestar-项目说明书.md` 第五章或 `src/setup.ts` 的引导文案。

### 7.2 本机部署现状(这台机器)
- daemon 由 **launchd** 托管:label `com.supercc168.lodestar`,plist `~/Library/LaunchAgents/com.supercc168.lodestar.plist`。
- 全局命令是 symlink 指向仓库 `dist/lodestar.js` —— **重新 `bun run build` 后 `launchctl kickstart -k gui/501/com.supercc168.lodestar` 即生效**,无需重装。
- Claude 走 **reclaude** 调度(`[claude] bin = "~/.local/bin/reclaude"`),日志出现 `executable=config:…/reclaude` 即生效。
- KeepAlive:崩溃自动拉起;重启后上次活跃 session 并发 `--resume` 接回。

> ⚠️ 上面的 `kickstart` 重启是 **live-service 操作**,受第 4 章红线一约束 —— 除非用户当条消息明确要求,否则只报告"需要重启",不主动执行。

## 8. 发布流程 + 常见坑

### 发布 checklist(以根 `AGENTS.md` 的 Release Checklist 为准)
1. 默认只 patch 版本 +0.0.1(除非用户要 minor/major)。
2. `bun test` + `bun run build` 验证。
3. 提交 release bump → 建 `vX.Y.Z` tag → push `main` → push tag。
4. `npm publish --access public`,并**同版本发布到 GitHub Packages**(临时 `.npmrc` + `@leviyuan:registry`,发完删)。
5. **务必**为 tag 建 GitHub Release(本机无 `gh`,走 REST API,token 从 `~/.git-credentials` 读);release notes 用中文、优先写"用户能感受到什么"、匹配既有风格。

### 常见坑
| 症状 | 原因 |
| --- | --- |
| daemon 起不来 | `config.toml` 缺 `[feishu].app_id/app_secret` |
| 群发消息无反应 | 应用没发布版本 / 缺 `im:message.group_msg` / 事件订阅没选长连接 |
| `401 Invalid bearer token` | 配了 reclaude 但没写 `[claude] bin`,或凭据不对 |
| 卡片一直 `Thinking...` + `model=<synthetic>` | `[projects.*]` 配一半(`.mcp.json` 缺失或 MCP 起不来);注释掉该节重启 |
| 测试 `bun:test` 报错为 vitest | 本项目用 `bun test`,不是 `npx vitest` |
